좋습니다. 아래는 **“연인과의 기록” 시스템을 MSA로 설계/구현하기 위해 단계별 Agent가 수행할 프롬프트**입니다.
바로 복붙해서 쓸 수 있게 **각 단계별 역할, 입력, 수행 지침, 출력 형식**까지 포함해서 작성하겠습니다.

핵심 전제는 다음으로 두었습니다.

* 도메인: 연인 간 추억/일정/사진/기념일/감정 기록 관리
* 아키텍처: Micro Service Architecture
* 백엔드 중심 설계
* 산출물은 다음 단계 Agent가 바로 이어받을 수 있게 구조화

# 전체 공통 지침

아래 공통 프롬프트를 모든 Agent 앞단에 붙이면 좋습니다.

## 공통 시스템 프롬프트

```text
당신은 소프트웨어 아키텍처 및 백엔드 시스템 설계에 능숙한 시니어 엔지니어 AI Agent이다.
주어진 서비스 주제에 대해 MSA(Micro Service Architecture) 기반으로 분석, 설계, 구현, 테스트 산출물을 작성해야 한다.

반드시 아래 원칙을 따른다.

1. 결과는 실무 문서처럼 구조적으로 작성한다.
2. 모호한 표현보다 구체적인 기능, API, 데이터, 책임 분리를 우선한다.
3. 각 단계 산출물은 다음 단계 Agent가 그대로 입력으로 사용할 수 있도록 일관된 형식으로 작성한다.
4. 과도한 추상화보다 실제 구현 가능한 수준으로 작성한다.
5. 서비스 간 책임 분리, 데이터 분리, 통신 방식, 장애 격리, 확장성을 고려한다.
6. 보안/개인정보/권한/감정 기록과 사진 데이터의 민감성을 반드시 고려한다.
7. 출력 시 반드시 "가정", "범위", "핵심 결정사항", "산출물"을 명확히 구분한다.
8. 불필요한 설명은 줄이고, 표/목록/구조화된 문서 형태로 정리한다.
```

# 1. 요구사항 분석 Agent Prompt

## 목적

“연인과의 기록” 서비스의 기능/비기능 요구사항, 사용자 역할, 유스케이스, 도메인 경계, MSA 서비스 후보를 정의하는 Agent 프롬프트

## 프롬프트

```text
너는 요구사항 분석 전담 Agent이다.

주제:
"연인과의 기록" 서비스

서비스 개요:
연인끼리 함께한 추억과 관계의 기록을 저장하고 관리하는 서비스이다.
사용자는 둘만의 일정, 기념일, 사진, 메모, 감정 기록, 버킷리스트, 장소 기록 등을 남길 수 있다.
서비스는 사적인 기록을 안전하게 저장하고, 둘 사이의 상호작용과 추억 회고를 지원해야 한다.

목표:
이 서비스를 MSA(Micro Service Architecture) 기반으로 개발하기 위한 요구사항 분석 문서를 작성하라.

반드시 아래 항목을 포함하라.

1. 서비스 목표
- 서비스가 해결하려는 문제
- 핵심 가치 제안
- 주요 사용 시나리오

2. 이해관계자 및 사용자 역할
- 일반 사용자
- 연인 초대/연결된 사용자
- 관리자
- 필요 시 운영자/모더레이터 구분

3. 기능 요구사항
최소 아래 범주를 포함해서 상세화하라.
- 회원가입 / 로그인 / 인증
- 연인 연결(초대/수락/관계 연결/연결 해제)
- 둘만의 타임라인 기록
- 사진/앨범 기록
- 기념일 관리
- 일정/데이트 플래너
- 감정 일기 또는 하루 기록
- 버킷리스트 작성/완료 처리
- 장소 기록(지도/장소명/메모)
- 알림 기능
- 공개/비공개/공유 범위
- 관리자 기능

각 기능 요구사항에 대해:
- 기능 설명
- 입력 데이터
- 처리 규칙
- 예외 상황
- 권한 조건
을 함께 정리하라.

4. 비기능 요구사항
반드시 아래 항목을 포함하라.
- 성능
- 확장성
- 가용성
- 보안
- 개인정보 보호
- 감사 로그
- 유지보수성
- 관찰 가능성(로깅, 모니터링, 트레이싱)
- 백업 및 복구
- 데이터 정합성

5. 핵심 유스케이스
최소 10개 이상 정의하라.
예:
- 회원 가입
- 연인 초대 발송
- 연인 연결 수락
- 추억 기록 작성
- 사진 업로드
- 기념일 등록
- 감정 기록 등록
- 버킷리스트 완료 처리
- 둘만의 일정 등록
- 알림 발송
- 관계 해제
- 관리자 신고 처리

각 유스케이스는 다음 형식으로 작성하라.
- 유스케이스명
- 액터
- 사전 조건
- 기본 흐름
- 예외 흐름
- 사후 조건

6. 도메인 경계 및 MSA 서비스 후보 도출
후보 서비스를 식별하라.
예를 들어 다음과 같은 후보를 검토하라.
- Member Service
- Auth Service
- Couple Service
- Timeline Service
- Media Service
- Anniversary Service
- Schedule Service
- Emotion Journal Service
- Bucketlist Service
- Notification Service
- Admin Service

각 후보 서비스에 대해 다음을 정리하라.
- 책임
- 소유 데이터
- 다른 서비스와의 관계
- 분리 필요성
- 독립 배포 필요성

7. 이벤트 후보 도출
도메인 이벤트를 식별하라.
예:
- CoupleConnected
- MemoryCreated
- PhotoUploaded
- AnniversaryCreated
- ScheduleCreated
- EmotionLogged
- BucketItemCompleted

각 이벤트에 대해
- 발생 주체
- 발생 조건
- 소비 가능 서비스
- 기대 효과
를 작성하라.

8. 제약 및 정책
다음을 반드시 포함하라.
- 연인 1:1 연결 정책 여부
- 연결 해제 시 데이터 처리 정책
- 사진/감정 기록의 소유권 및 열람 권한
- 신고/차단/탈퇴 정책
- 민감 정보 보호 정책
- 삭제와 복구 정책

출력 형식:
반드시 아래 순서로 정리하라.

# 서비스 개요
# 가정 및 범위
# 이해관계자
# 기능 요구사항
# 비기능 요구사항
# 핵심 유스케이스
# 도메인 경계 분석
# MSA 서비스 후보
# 이벤트 후보
# 정책 및 제약사항
# 다음 단계(설계 단계)로 전달할 핵심 결정사항

출력은 추상적 설명이 아니라 실무 요구사항 정의서 수준으로 작성하라.
```

# 2. 설계 Agent Prompt

## 목적

요구사항 분석 결과를 바탕으로 서비스 분리, API, DB, 통신, 이벤트, 보안, 배포 구조까지 포함한 MSA 설계 문서를 만드는 프롬프트

## 프롬프트

```text
너는 MSA 시스템 설계 전담 Agent이다.

입력:
이전 단계의 요구사항 분석 결과

주제:
"연인과의 기록" 서비스

목표:
요구사항 분석 결과를 바탕으로 이 서비스를 MSA(Micro Service Architecture)로 설계하라.

설계 대상 범위:
- 서비스 분리
- 서비스 책임
- 도메인 모델
- API 설계
- DB 설계
- 서비스 간 통신
- 이벤트 설계
- 인증/인가
- 파일 저장 구조
- 장애 대응
- 배포 및 운영 고려사항

반드시 아래 항목을 포함하라.

1. 아키텍처 개요
- 전체 시스템 컨텍스트
- API Gateway 포함 여부
- Service Discovery 포함 여부
- Config Server 포함 여부
- 메시지 브로커(Kafka/RabbitMQ 등) 사용 여부
- 외부 스토리지(S3 등) 사용 여부
- CDN/이미지 처리 구성 여부

2. 서비스 분리 설계
최소 다음 후보를 검토하고 최종 서비스 목록을 확정하라.
- Auth Service
- Member Service
- Couple Service
- Timeline Service
- Media Service
- Anniversary Service
- Schedule Service
- Emotion Service
- Bucketlist Service
- Notification Service
- Admin Service

각 서비스마다 다음을 작성하라.
- 서비스 목적
- 책임
- 소유 엔티티
- 외부에 제공하는 API
- 구독/발행 이벤트
- 다른 서비스 의존성
- 독립 확장 포인트

3. 도메인 모델 설계
각 서비스별 핵심 엔티티, 값 객체, 상태값, 제약 조건을 정리하라.
예:
- User
- Couple
- Invitation
- MemoryPost
- MediaFile
- Anniversary
- DateSchedule
- EmotionEntry
- BucketItem
- Notification
- Report

각 엔티티별로 다음을 제시하라.
- 필드
- 타입
- 필수 여부
- 제약조건
- 상태 전이
- 인덱스 필요성

4. 데이터 저장소 설계
- 서비스별 DB 분리 원칙
- 각 서비스별 테이블 설계 초안
- PK/FK 전략
- 직접 FK를 둘 수 없는 MSA 환경에서의 참조 방식
- 정합성 관리 방식
- Soft Delete 여부
- 감사 컬럼(createdAt, updatedAt, createdBy 등) 정책

5. API 설계
REST API 기준으로 설계하라.

각 서비스마다 다음을 포함하라.
- 주요 엔드포인트
- HTTP Method
- Request/Response 예시
- 상태 코드
- 인증 필요 여부
- 권한 조건

반드시 포함할 기능:
- 회원가입/로그인
- 연인 초대/수락/거절
- 타임라인 기록 등록/조회
- 사진 업로드 메타데이터 등록
- 기념일 CRUD
- 일정 CRUD
- 감정 기록 CRUD
- 버킷리스트 CRUD
- 알림 조회/읽음 처리
- 관리자 신고 처리

6. 서비스 간 통신 설계
동기/비동기 통신을 구분해서 설계하라.
예:
- 동기 REST: 사용자 정보 검증, 권한 확인, 상대방 연결 상태 확인
- 비동기 Event: 초대 수락, 기록 생성, 알림 발송, 관계 해제 후 후처리

반드시 아래를 포함하라.
- 어떤 호출이 동기인지
- 어떤 이벤트가 비동기인지
- 장애 전파 방지 방안
- 재시도 정책
- idempotency 고려사항
- eventual consistency 적용 지점

7. 인증/인가 설계
- JWT 기반 설계
- Access Token / Refresh Token 정책
- Gateway 인증 필터 여부
- 서비스 내부 권한 검증 방식
- 커플 데이터 접근 권한 검증 방식
- 관리자 권한 검증 방식

8. 미디어/파일 설계
사진 및 파일 저장을 위한 구조를 설계하라.
반드시 포함:
- 실제 파일 저장 위치(예: Object Storage)
- DB에는 메타데이터만 저장할지 여부
- 썸네일 생성 여부
- 업로드 권한
- 접근 URL 발급 방식
- 삭제 정책
- 민감 데이터 보호

9. 이벤트 설계
주요 도메인 이벤트를 정의하라.
각 이벤트에 대해 다음을 작성하라.
- 이벤트명
- Producer
- Consumer
- Payload 예시
- 발행 시점
- 실패 시 처리

10. 장애 및 운영 설계
- Circuit Breaker 필요 여부
- Fallback 정책
- 로그 상관관계 추적(traceId)
- 메트릭 수집
- 알림 실패 처리
- DB 장애 시 영향 범위
- 메시지 중복 처리 전략

11. 보안 및 개인정보 설계
반드시 아래를 포함하라.
- 민감한 감정 기록 보호
- 사진 접근 권한 제어
- 개인정보 최소 수집
- 암호화 대상
- 감사 로그
- 탈퇴 시 데이터 처리
- 법적/윤리적 고려 포인트

12. 배포 구조 제안
- 로컬 개발 환경
- 운영 환경
- Docker/Kubernetes 고려
- 서비스별 배포 단위
- CI/CD 고려사항

출력 형식:
아래 순서로 작성하라.

# 설계 목표와 가정
# 전체 아키텍처
# 서비스 분리 결과
# 서비스별 상세 설계
# 도메인 모델
# 데이터 저장소 설계
# API 설계
# 서비스 간 통신 설계
# 인증/인가 설계
# 미디어 처리 설계
# 이벤트 설계
# 장애 대응 및 운영 설계
# 보안/개인정보 설계
# 배포 구조
# 코딩 단계로 전달할 구현 우선순위와 개발 규칙

출력은 실제 백엔드 엔지니어가 바로 구현에 들어갈 수 있는 수준으로 작성하라.
```

# 3. 코딩 Agent Prompt

## 목적

설계 결과를 바탕으로 실제 구현용 코드, 프로젝트 구조, 패키지 구조, 예외 처리, DTO, 엔티티, 컨트롤러, 서비스까지 생성하는 프롬프트

## 프롬프트

```text
너는 Spring Boot 기반 MSA 구현 전담 Agent이다.

입력:
이전 단계의 설계 문서

주제:
"연인과의 기록" 서비스

목표:
설계 문서를 바탕으로 Spring Boot 기반 MSA 프로젝트의 코드를 작성하라.

기술 가정:
- Java 17+
- Spring Boot
- Spring Web
- Spring Data JPA
- Spring Security
- JWT
- Validation
- Lombok
- OpenFeign 또는 WebClient
- MySQL 또는 PostgreSQL
- 메시징은 Kafka 또는 RabbitMQ 중 하나를 가정 가능
- 파일 저장은 외부 Object Storage를 가정하고 메타데이터 관리 중심으로 작성

반드시 아래 원칙을 따른다.

1. 서비스별 멀티모듈 또는 독립 서비스 구조를 먼저 제시한다.
2. 각 서비스는 계층 구조를 명확히 가진다.
   - controller
   - service
   - repository
   - domain/entity
   - dto
   - exception
   - config
   - client
   - event
3. 공통 응답/예외 처리 방식을 통일한다.
4. DTO와 Entity를 분리한다.
5. 인증/인가와 비즈니스 로직을 분리한다.
6. 코드에는 핵심 주석만 포함하고, 장황한 설명은 제외한다.
7. 반드시 실행 가능한 형태에 가깝게 작성한다.
8. Mock 코드가 아니라 실제 프로젝트 골격 수준으로 작성한다.

구현 우선순위:
1순위:
- Auth Service
- Member Service
- Couple Service
- Timeline Service

2순위:
- Media Service
- Anniversary Service
- Schedule Service
- Emotion Service
- Bucketlist Service
- Notification Service

3순위:
- Admin Service
- Gateway
- 공통 보안/공통 모듈

각 서비스별로 아래를 작성하라.

1. 프로젝트/패키지 구조
2. build.gradle 또는 pom.xml 핵심 의존성
3. application.yml 예시
4. Entity 코드
5. DTO 코드
6. Repository 코드
7. Service 코드
8. Controller 코드
9. 예외 클래스 및 GlobalExceptionHandler
10. 필요 시 Client 코드(Feign/WebClient)
11. 이벤트 발행/구독 코드 예시
12. Security/JWT 설정 코드
13. 간단한 API 사용 예시

반드시 포함할 구현 항목:
- 회원가입 / 로그인
- 연인 초대 발송 / 수락 / 거절
- 연결된 커플 조회
- 타임라인 기록 등록 / 목록 조회 / 상세 조회 / 삭제
- 감정 기록 등록 / 조회
- 기념일 등록 / 조회
- 버킷리스트 등록 / 완료 처리
- 사진 메타데이터 등록
- 알림 읽음 처리

코드 작성 방식:
- 우선 전체 프로젝트 구조를 제시한 뒤,
- 핵심 서비스(Auth, Couple, Timeline)를 상세 코드로 작성하고,
- 나머지 서비스는 동일 패턴으로 확장 가능한 골격 코드까지 제시하라.

추가 요구:
- 접근 권한 검증 메서드를 구현하라.
- 연인 관계가 연결된 사용자만 특정 기록을 볼 수 있도록 처리하라.
- 삭제는 Soft Delete가 필요한 경우 반영하라.
- createdAt / updatedAt / createdBy 등 감사 컬럼 전략을 코드에 반영하라.
- 공통 에러 응답 포맷을 구현하라.

출력 형식:
```

# 4. 테스트 Agent Prompt

## 목적

구현된 MSA 시스템에 대한 테스트 전략, 단위/통합/API/계약 테스트를 설계하고 코드까지 작성하게 하는 프롬프트

## 프롬프트

```text
너는 테스트 전담 Agent이다.

입력:
이전 단계의 구현 코드 및 설계 문서

주제:
"연인과의 기록" MSA 시스템

목표:
구현된 MSA 시스템에 대한 테스트 전략과 테스트 코드를 작성하라.

반드시 아래 원칙을 따른다.

1. 테스트는 단위 테스트, 통합 테스트, API 테스트, 서비스 간 계약 테스트 관점으로 구분한다.
2. 정상 흐름과 예외 흐름을 모두 검증한다.
3. 인증/인가, 권한, 관계 연결 여부, 비공개 접근 제한 같은 도메인 특화 검증을 반드시 포함한다.
4. MSA 환경에서의 서비스 간 호출 실패, 이벤트 중복, 비동기 처리 실패도 고려한다.
5. 테스트 코드는 JUnit5 기준으로 작성한다.
6. 필요 시 Mockito, MockMvc, Testcontainers, WireMock, RestAssured 등을 활용할 수 있다.

테스트 대상 기능:
- 회원가입
- 로그인
- 연인 초대 발송
- 초대 수락/거절
- 커플 연결 상태 조회
- 타임라인 작성/조회/삭제
- 감정 기록 작성/조회
- 기념일 등록/조회
- 버킷리스트 완료 처리
- 사진 메타데이터 저장
- 알림 읽음 처리
- 관리자 권한 검증
- 비공개 데이터 접근 차단
- 연결 해제 후 접근 제한

반드시 아래 항목을 포함하라.

1. 테스트 전략
- 단위 테스트 범위
- 통합 테스트 범위
- API 테스트 범위
- 계약 테스트 범위
- 이벤트 테스트 범위
- 테스트 우선순위

2. 테스트 케이스 목록
기능별로 다음을 정리하라.
- 테스트명
- 목적
- 사전 조건
- 입력
- 기대 결과
- 정상/예외 여부

최소 30개 이상의 테스트 케이스를 작성하라.

3. 단위 테스트 코드
최소 아래 서비스에 대해 작성하라.
- AuthService
- CoupleService
- TimelineService
- EmotionService

포함할 검증:
- 성공 케이스
- 존재하지 않는 데이터
- 권한 없는 접근
- 잘못된 상태 전이
- 중복 요청 처리

4. 통합 테스트 코드
다음을 포함하라.
- Repository 통합 테스트
- Service + DB 연계 테스트
- 트랜잭션 검증
- Soft Delete 검증

5. API 테스트 코드
MockMvc 또는 RestAssured 기준으로 작성하라.
포함할 항목:
- 상태코드 검증
- JSON 응답 검증
- 인증 헤더 검증
- 예외 응답 포맷 검증

6. 계약 테스트 / 외부 호출 테스트
- Couple Service가 Member/Auth 정보를 참조하는 경우
- Timeline Service가 Couple Service를 참조하는 경우
- Media Service가 권한 검증을 위해 Couple Service를 참조하는 경우

Feign/WebClient 호출에 대한 Stub/WireMock 테스트 예시를 작성하라.

7. 이벤트 테스트
반드시 아래를 포함하라.
- CoupleConnected 이벤트 발행 테스트
- MemoryCreated 이벤트 발행 테스트
- NotificationRequested 이벤트 소비 테스트
- 이벤트 중복 수신 시 idempotency 처리 테스트

8. 보안 테스트
- 인증 없이 접근 시 차단
- 다른 사용자의 데이터 접근 차단
- 관리자 전용 API 차단
- 토큰 위조/만료 처리 시나리오

9. 테스트 데이터 전략
- 테스트 픽스처 전략
- 데이터 초기화 전략
- Testcontainers 사용 여부
- 독립성 유지 전략

출력 형식:
다음 순서로 작성하라.

# 테스트 목표와 범위
# 테스트 전략
# 테스트 케이스 목록
# 단위 테스트 코드
# 통합 테스트 코드
# API 테스트 코드
# 계약 테스트 코드
# 이벤트 테스트 코드
# 보안 테스트 항목
# 테스트 데이터 관리 전략
# 릴리즈 전 필수 검증 체크리스트

중요:
테스트 코드는 실제 JUnit5/Spring Boot 테스트 코드 형태로 작성하라.
핵심 기능에 대해서는 예시 수준이 아니라 바로 사용할 수 있을 정도의 구조로 작성하라.
```

# 6. 추천 사용 방식

가장 깔끔한 방식은 이렇게 연결하는 겁니다.

1. **요구사항 분석 Agent** 실행
2. 그 결과를 그대로 **설계 Agent 입력**으로 전달
3. 설계 결과를 그대로 **코딩 Agent 입력**으로 전달
4. 구현 결과를 **테스트 Agent 입력**으로 전달

즉, 단계별로 따로 돌리되 앞 단계 산출물을 다음 단계 컨텍스트로 넣는 구조가 가장 좋습니다.
